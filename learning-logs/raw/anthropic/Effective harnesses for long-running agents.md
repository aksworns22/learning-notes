
> 우리는 [Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview)가 여러 컨텍스트 창에 걸쳐 효과적으로 작업할 수 있도록 두 갈래의 해결책을 개발했습니다. 첫 실행 때 환경을 설정하는 초기화 에이전트와, 매 세션마다 점진적으로 진척을 내면서 다음 세션을 위해 명확한 산출물을 남기는 코딩 에이전트입니다.

## The long-running agent problem

AI 에이전트의 역량이 커지면서, 개발자들은 몇 시간, 심지어 며칠에 걸친 작업이 필요한 복잡한 과제를 에이전트에게 점점 더 많이 맡기고 있습니다.

하지만 여러 컨텍스트 창에 걸쳐 에이전트가 일관되게 진척을 내도록 하는 일은 여전히 해결되지 않은 문제로 남아 있습니다.
- 장시간 실행되는 에이전트의 핵심 과제는 이들이 분리된 세션 단위로 작업해야 하며, 새 세션이 시작될 때마다 이전에 있었던 일을 기억하지 못한 채 시작한다는 점입니다.

Claude Agent SDK는 강력한 범용 에이전트 하네스로, 코딩뿐 아니라 모델이 도구를 사용해 컨텍스트를 수집하고 계획하며 실행해야 하는 다른 작업에도 능숙합니다.
- 이 SDK에는 압축(compaction) 같은 컨텍스트 관리 기능이 있어, 에이전트가 컨텍스트 창을 소진하지 않고 작업을 계속할 수 있게 해 줍니다. 
 
 > 압축만으로는 충분하지 않다!
 
 Claude의 실패는 두 가지 패턴으로 나타났습니다.
 - 첫째, 에이전트는 한 번에 너무 많은 일을 하려는 경향이 있었습니다. 본질적으로 앱을 한 번에 완성하려고 시도한 것입니다.
 - 두 번째 실패 양상은 프로젝트 후반에 자주 발생했습니다.몇몇 기능이 이미 만들어진 뒤, 나중에 실행된 에이전트 인스턴스가 주변을 살펴보고 진척이 있었다는 사실만 확인한 뒤 작업이 끝났다고 선언하곤 했습니다.

이 두 문제를 각각 바라볼 수 있다.
- 첫째, 주어진 프롬프트가 요구하는 모든 기능의 기반을 마련하는 초기 환경을 설정해야 합니다. 이를 통해 에이전트가 단계별로, 기능별로 작업할 수 있게 됩니다.
- 둘째, 각 에이전트가 목표를 향해 점진적으로 진척을 내도록 프롬프트를 주는 동시에, 세션이 끝날 때 환경을 깨끗한 상태로 남기게 해야 합니다.
	- 여기서 “깨끗한 상태”란 메인 브랜치에 병합해도 적절한 수준의 코드를 뜻합니다. 중대한 버그가 없고, 코드가 정돈되어 있으며 잘 문서화되어 있고, 일반적으로 개발자가 관련 없는 문제를 먼저 정리하지 않고도 새 기능 작업을 쉽게 시작할 수 있는 상태입니다.

실제 내부 실험

> 핵심 통찰은 에이전트가 새로운 컨텍스트 창으로 시작할 때 작업 상태를 빠르게 이해할 수 있는 방법을 찾는 것이었고, 이는 git 히스토리와 함께 `claude-progress.txt` 파일을 두는 방식으로 달성됩니다.


초기화 에이전트: 첫 번째 에이전트 세션은 모델에게 초기 환경을 설정하라고 요청하는 특수 프롬프트를 사용합니다. 여기에는 `init.sh` 스크립트, 에이전트들이 수행한 일을 기록하는 `claude-progress.txt` 파일, 그리고 어떤 파일이 추가되었는지 보여 주는 초기 git 커밋이 포함됩니다.

코딩 에이전트: 이후의 모든 세션은 모델에게 점진적인 진척을 내고, 그다음 구조화된 업데이트를 남기도록 요청합니다.

## Environment management

### Feature list

코딩 에이전트들이 효과적으로 작업하는 데 필요한 모든 컨텍스트를 갖춘 환경을 구성하는 몇 가지 핵심 요소

에이전트가 앱을 한 번에 완성하려 하거나 프로젝트가 완료되었다고 성급하게 판단하는 문제를 해결하기 위해, 우리는 초기화 에이전트에게 사용자의 초기 프롬프트를 확장한 포괄적인 기능 요구사항 파일을 작성하도록 프롬프트를 주었습니다.

우리는 코딩 에이전트에게 이 파일을 수정할 때 `passes` 필드의 상태만 변경하라고 프롬프트를 주며, “테스트를 제거하거나 수정하는 것은 누락된 기능이나 버그 있는 기능으로 이어질 수 있으므로 용납할 수 없다”와 같은 강한 표현의 지시를 사용합니다.
### Incremental progress

초기 환경 스캐폴딩이 주어진 뒤, 다음 버전의 코딩 에이전트에게는 한 번에 하나의 기능만 작업하라고 요청했습니다.

이 점진적 접근법은 에이전트가 한 번에 너무 많은 일을 하려는 경향을 해결하는 데 핵심적인 역할을 했습니다.
- 점진적으로 작업하더라도, 모델이 코드 변경을 한 뒤 환경을 깨끗한 상태로 남기는 것은 여전히 필수적입니다.

우리의 실험에서, 이러한 행동을 이끌어내는 가장 좋은 방법은 모델에게 설명적인 커밋 메시지와 함께 진행 상황을 git에 커밋하고, 진행 요약을 진행 파일에 작성하라고 요청하는 것임을 발견했습니다.
- 이러한 접근법은 효율성도 높였습니다. 에이전트가 무슨 일이 있었는지 추측하고 기본 앱을 다시 동작시키는 데 시간을 써야 할 필요를 없앴기 때문입니다.

### Testing

우리가 관찰한 마지막 주요 실패 양상은 Claude가 적절한 테스트 없이 기능을 완료된 것으로 표시하는 경향이었습니다.

명시적인 프롬프트가 없으면 Claude는 코드 변경을 하고, 개발 서버에 대해 단위 테스트나 `curl` 명령으로 테스트까지 수행하는 경향이 있었지만, 해당 기능이 엔드투엔드로 동작하지 않는다는 사실을 알아차리지 못하곤 했습니다.
- 웹 앱을 만드는 경우, Claude는 브라우저 자동화 도구를 사용하고 인간 사용자가 하듯 모든 테스트를 수행하라는 명시적인 프롬프트를 받은 뒤에는 대체로 기능을 엔드투엔드로 잘 검증했습니다.

## Getting up to speed

위의 모든 것이 갖춰진 상태에서, 모든 코딩 에이전트는 상황을 파악하기 위해 일련의 단계를 수행하도록 프롬프트를 받습니다. 그중 일부는 꽤 기본적이지만 여전히 유용합니다.
- pwd실행
- git log와 진행 파일 읽기
- 기능 목록 파일을 읽고, 아직 완료되지 않은 기능 중 우선순위가 가장 높은 것을 선택 / 작업

이 접근법은 Claude가 코드를 어떻게 테스트해야 할지 매 세션마다 알아낼 필요가 없게 해 주므로 토큰을 절약합니다.

또한 초기화 에이전트에게 개발 서버를 실행할 수 있는 `init.sh` 스크립트를 작성하게 하고, 새 기능을 구현하기 전에 기본적인 엔드투엔드 테스트를 수행하게 하는 것도 도움이 됩니다.
- 이를 통해 Claude는 앱이 망가진 상태로 남아 있는지 빠르게 확인하고, 기존 버그를 즉시 수정할 수 있었습니다.만약 에이전트가 대신 새 기능 구현부터 시작했다면, 문제를 더 악화시켰을 가능성이 큽니다.

## Future work

하나의 범용 코딩 에이전트가 여러 컨텍스트에 걸쳐 가장 좋은 성능을 내는지, 아니면 다중 에이전트 아키텍처를 통해 더 나은 성능을 얻을 수 있는지는 아직 명확하지 않습니다.
- 테스트 에이전트, 품질 보증 에이전트, 코드 정리 에이전트 같은 전문화된 에이전트가 소프트웨어 개발 생애주기의 하위 작업들을 더 잘 수행할 수 있을 것이라고 보는 것은 타당해 보입니다.

또한 이 데모는 풀스택 웹 앱 개발에 최적화되어 있습니다.
- 향후 방향은 이러한 발견을 다른 분야로 일반화하는 것입니다.
