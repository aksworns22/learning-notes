## 장기 실행 에이전트를 위한 효과적인 하네스

### 문제: 컨텍스트 창을 넘나드는 일관된 진척

AI 에이전트가 몇 시간, 며칠씩 걸리는 복잡한 작업을 맡게 되면서, 여러 컨텍스트 창에 걸쳐 일관되게 진척을 내는 것이 핵심 과제로 떠올랐다.

- 장기 실행 에이전트는 분리된 세션 단위로 작업하며, 새 세션이 시작될 때마다 이전에 무슨 일이 있었는지 기억하지 못한 채 시작한다.
- Claude Agent SDK는 압축(compaction) 같은 컨텍스트 관리 기능을 제공해 컨텍스트 창 소진을 막아주지만, **압축만으로는 충분하지 않다.**

실제로 관찰된 두 가지 실패 패턴:

1. 에이전트가 한 번에 너무 많은 일을 하려고 함 — 앱을 한 번에 완성하려 시도.
2. 프로젝트 후반에, 이미 일부 기능이 만들어진 뒤 실행된 에이전트 인스턴스가 주변을 둘러보고 진척이 있었다는 사실만 확인한 채 작업이 끝났다고 성급히 선언함.

이 두 문제에 대응하기 위해 두 갈래 해법을 마련했다:

- **초기화 에이전트**: 첫 세션에서 프롬프트가 요구하는 모든 기능의 기반이 되는 환경을 설정.
- **코딩 에이전트**: 이후 모든 세션에서 목표를 향해 점진적으로 진척을 내고, 세션이 끝날 때 메인 브랜치에 병합 가능한 "깨끗한 상태"(버그 없음, 정돈된 코드, 잘 문서화됨)로 환경을 남김.

핵심 통찰은 새 컨텍스트 창으로 시작하는 에이전트가 작업 상태를 빠르게 이해할 수 있어야 한다는 것이며, 이를 git 히스토리와 `claude-progress.txt` 파일로 달성했다.

### 환경 관리

#### 기능 목록 (Feature list)

에이전트가 한 번에 너무 많은 일을 하려 하거나 프로젝트가 끝났다고 성급히 판단하는 문제를 막기 위해, 초기화 에이전트에게 사용자의 초기 프롬프트를 확장한 포괄적인 기능 요구사항 파일을 작성하게 했다.

- 코딩 에이전트에게는 이 파일을 수정할 때 `passes` 필드의 상태만 바꾸도록 강하게 제약했다. ("테스트를 제거하거나 수정하는 것은 누락된 기능이나 버그 있는 기능으로 이어질 수 있으므로 용납할 수 없다.")

#### 점진적 진척 (Incremental progress)

초기 환경 스캐폴딩이 갖춰진 뒤, 코딅 에이전트에게는 한 번에 하나의 기능만 작업하도록 요청했다. 이 접근이 "한 번에 너무 많은 일을 하려는" 실패 패턴을 해결하는 핵심이었다.

- 점진적으로 작업해도, 코드 변경 후 환경을 깨끗한 상태로 남기는 것은 여전히 필수.
- 가장 효과적인 방법: 설명적인 커밋 메시지와 함께 git에 커밋하고, 진행 요약을 progress 파일에 남기도록 요청하는 것.
- 이 방식은 다음 에이전트가 무슨 일이 있었는지 추측하고 앱을 다시 동작시키는 데 시간을 쓸 필요를 없애 효율성도 높였다.

#### 테스트 (Testing)

마지막 주요 실패 양상은 Claude가 적절한 테스트 없이 기능을 완료로 표시하는 경향이었다.

- 명시적 프롬프트가 없으면 Claude는 단위 테스트나 `curl` 테스트는 수행해도, 기능이 엔드투엔드로 동작하지 않는다는 사실은 놓치곤 했다.
- 웹 앱의 경우, 브라우저 자동화 도구로 인간 사용자처럼 직접 테스트하라는 명시적 프롬프트를 줬을 때 엔드투엔드 검증 품질이 크게 좋아졌다.

### 세션 시작 시 상황 파악

위 요소들이 갖춰진 상태에서, 모든 코딩 에이전트는 세션 시작 시 다음 단계를 거치도록 프롬프트를 받는다.

- `pwd` 실행
- `git log`와 progress 파일 읽기
- 기능 목록 파일을 읽고, 미완료 기능 중 우선순위가 가장 높은 것을 선택해 작업

이 루틴은 Claude가 매 세션마다 코드를 어떻게 테스트할지 새로 알아낼 필요를 없애 토큰을 절약한다.

또한 초기화 에이전트에게 개발 서버를 실행하는 `init.sh` 스크립트를 작성하게 하고, 새 기능 구현 전에 기본적인 엔드투엔드 테스트를 먼저 수행하게 한 것도 도움이 되었다.

- 이를 통해 앱이 망가진 상태로 남아있는지 빠르게 확인하고 기존 버그를 즉시 고칠 수 있었다. 반대로 새 기능 구현부터 시작했다면 문제를 더 악화시켰을 가능성이 크다.

### 향후 방향

- 하나의 범용 코딩 에이전트가 여러 컨텍스트에 걸쳐 가장 좋은 성능을 내는지, 아니면 테스트 에이전트·QA 에이전트·코드 정리 에이전트 같은 전문화된 다중 에이전트 아키텍처가 소프트웨어 개발 생애주기의 하위 작업을 더 잘 수행하는지는 아직 불명확하다.
- 이번 실험은 풀스택 웹 앱 개발에 최적화되어 있어, 다른 분야로 일반화하는 것이 향후 과제다.
