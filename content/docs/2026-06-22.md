## 임베딩과 벡터 유사도 검색


<p>GraphRAG를 다루기 전에, 기존 벡터 기반 RAG가 어떻게 동작하고 어디서 한계를 보이는지부터 확인했다. 출발점은 임베딩이다. 텍스트를 <code>OpenAIEmbeddings</code>(<code>text-embedding-3-small</code>)로 변환하면 의미가 담긴 고차원 벡터(1536차원)가 나오고, 두 벡터의 코사인 유사도로 "의미가 얼마나 비슷한지"를 수치화할 수 있다.</p>

<ul>
  <li>코사인 유사도는 <code>(v1 · v2) / (|v1| * |v2|)</code>로 계산한다 — 벡터의 방향(의미)이 얼마나 일치하는지를 보고, 크기(텍스트 길이 등)는 무시한다.</li>
  <li>날씨 관련 문장들끼리는 유사도가 높고, 프로그래밍 관련 문장과는 유사도가 낮게 나오는 것을 직접 확인했다.</li>
  <li>벡터스토어(<code>InMemoryVectorStore</code>)를 쓰면 이 유사도 계산과 정렬(top-k 추출)을 직접 구현할 필요 없이 <code>similarity_search</code> 한 줄로 처리된다.</li>
  <li>흥미로운 점: 한국어 문서 집합에서 영어 질문(<code>"Who makes Korean Language"</code>)을 던져도 관련 문서가 검색됐다 — 임베딩 모델이 언어를 넘어 의미 공간에서 매칭하기 때문.</li>
</ul>

<div class="mermaid">
graph LR
  q["질문/문서 텍스트"] --> e["임베딩 모델<br>(text-embedding-3-small)"]
  e --> v["벡터"]
  v -->|코사인 유사도| s["유사도 점수"]
  s -->|상위 k개| r["검색 결과"]
</div>

<details>
<summary>코사인 유사도 직접 계산하는 코드</summary>
<pre><code>def cosine_similarity(vec1, vec2):
    vec1 = np.array(vec1)
    vec2 = np.array(vec2)
    return np.dot(vec1, vec2) / (np.linalg.norm(vec1) * np.linalg.norm(vec2))

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
doc_vectors = embeddings.embed_documents(documents)

def search(query: str, top_k: int = 3) -> list:
    query_vector = embeddings.embed_query(query)
    similarities = [(i, cosine_similarity(query_vector, v)) for i, v in enumerate(doc_vectors)]
    similarities.sort(key=lambda x: x[1], reverse=True)
    return [{"document": documents[i], "similarity": s} for i, s in similarities[:top_k]]
</code></pre>
</details>

## 텍스트 청킹


<p>긴 문서를 그대로 임베딩하면 의미가 희석되므로, 임베딩 전에 적절한 크기로 잘라야 한다. <code>RecursiveCharacterTextSplitter</code>는 <code>chunk_size</code>(청크 최대 길이)와 <code>chunk_overlap</code>(청크 간 겹치는 길이)을 지정해 문단·문장 단위를 최대한 보존하면서 분할한다. 겹치는 구간을 두는 이유는 청크 경계에서 문맥이 끊기는 것을 완화하기 위함이다.</p>

## 벡터 검색 기반 RAG와 그 한계


<p>세종대왕, 정조, 이순신, 집현전에 대한 한국사 문서 6개를 청킹 → 임베딩 → 벡터스토어 저장 → <code>retriever</code> + LLM 체인으로 묶어 표준적인 RAG 파이프라인을 구성했다.</p>

<div class="mermaid">
graph LR
  docs["문서들"] --> split["청킹"]
  split --> embed["임베딩"]
  embed --> store["벡터 스토어"]
  q["질문"] --> retriever["retriever<br>(top-k 검색)"]
  store --> retriever
  retriever --> ctx["검색된 컨텍스트"]
  ctx --> prompt["프롬프트"]
  q --> prompt
  prompt --> llm["LLM"]
  llm --> ans["답변"]
</div>

<p>단순 사실 조회 질문("세종대왕이 발명한 것들은?")에는 잘 답했지만, 의도적으로 어려운 질문을 던져보니 한계가 분명했다.</p>

<ul>
  <li><strong>관계 추론</strong>: "세종대왕과 집현전의 관계는?" — 두 개체 사이의 연결고리를 벡터 유사도만으로는 명시적으로 찾기 어렵다.</li>
  <li><strong>다단계 추론</strong>: "세종대왕이 설치한 기관에서 누가 활동했나?" — "세종대왕 → 집현전 설치 → 집현전에서 활동한 인물"처럼 두 단계를 거쳐야 하는데, 벡터 검색은 질문 자체와 유사한 청크만 가져오기 때문에 중간 연결 단계를 놓치기 쉽다.</li>
  <li><strong>비교 질문</strong>: "세종대왕과 정조의 공통점은?" — 두 인물에 대한 청크를 각각 가져와도, 그 안에서 공통점을 뽑아내는 건 검색이 아니라 LLM의 추론에 전적으로 의존하게 된다.</li>
  <li><strong>시간순 나열</strong>: "조선의 왕들을 시대순으로 나열" — 문서에 시간 순서가 구조화되어 있지 않으면 검색만으로는 답할 수 없다.</li>
</ul>

<p>즉, 벡터 검색은 "질문과 의미적으로 비슷한 텍스트 조각"을 찾는 데는 강하지만, 개체(entity) 사이의 관계나 다단계 연결을 명시적으로 표현하지 못한다. 이 한계가 바로 GraphRAG의 동기다 — 문서를 청크가 아니라 개체와 관계로 이루어진 지식 그래프로 표현하면, "세종대왕 -[설치]-> 집현전 -[소속]-> 성삼문" 같은 경로를 그래프 탐색으로 직접 따라갈 수 있다.</p>

## Neo4j로 그래프 데이터베이스 다루기


<p>GraphRAG의 그래프 저장소로 쓰이는 Neo4j를 처음 연결해보고, 기본적인 노드/관계 생성과 조회를 익혔다.</p>

<ul>
  <li><code>GraphDatabase.driver(URI, auth=...)</code>로 드라이버를 만들고 <code>verify_connectivity()</code>로 연결을 확인한다.</li>
  <li>Cypher 쿼리로 노드(<code>CREATE (p:Person {...})</code>)와 그 사이의 관계(<code>CREATE (p)-[:WORKS_AT]->(c)</code>)를 한 번에 생성할 수 있다.</li>
  <li>조회도 패턴 매칭으로 한다: <code>MATCH (p:Person)-[:WORKS_AT]->(c:Company) RETURN ...</code> — "관계를 따라가는 질의"가 SQL의 JOIN보다 직접적으로 표현된다.</li>
</ul>

<div class="mermaid">
graph LR
  p["Person<br>김철수"] -->|WORKS_AT| c["Company<br>AI 스타트업"]
</div>

<details>
<summary>노드/관계 생성 및 조회 코드</summary>
<pre><code>with driver.session() as session:
    session.run("""
        CREATE (p: Person {name: '김철수', age: 30})
        CREATE (c: Company {name: 'AI 스타트업'})
        CREATE (p)-[:WORKS_AT]->(c)
    """)

    result = session.run("""
        MATCH (p:Person)-[:WORKS_AT]->(c:Company)
        RETURN p.name as person, c.name as company
    """)
    for record in result:
        print(f"{record['person']}님은 {record['company']}에서 일합니다.")
</code></pre>
</details>

<p>이 노드-관계 모델이 다음 단계의 핵심이다. 위에서 확인한 "다단계 추론", "관계 질문" 같은 한계를, 문서에서 추출한 개체와 관계를 Neo4j 그래프로 저장해두고 그래프 탐색으로 보완하는 것이 GraphRAG의 방향이다.</p>

## 언어 모델이 챗GPT가 되기까지: 스케일링과 정렬


<p>모델 크기와 학습 데이터가 커질수록 성능이 좋아지는 이유를, 학습 과정 자체를 <strong>데이터 압축</strong>으로 보는 관점에서 다시 짚었다. 언어 모델은 학습 데이터를 그대로 외우는 게 아니라, 데이터 안에서 공통되고 중요한 패턴만 남기는 손실 압축을 수행한다고 볼 수 있다. 이 관점에서는 학습 데이터의 크기가 모델이 의미 있게 키울 수 있는 최대 크기의 상한이 된다 — 데이터가 적은데 모델만 키우면 압축할 패턴이 부족해 과적합으로 이어지기 쉽다.</p>

<p>GPT-3 같은 사전 학습 모델은 "다음 단어 예측"만 학습했을 뿐, 사용자의 요청 의도에 맞게 답하도록 훈련되지는 않았다. LLM의 답변을 사용자 의도에 맞추는 과정을 <strong>정렬(alignment)</strong>이라고 부르고, GPT-3를 챗GPT로 바꾼 것이 바로 이 정렬 작업이다.</p>

<div class="mermaid">
graph LR
  pretrain["사전 학습<br>(다음 단어 예측)"] --> sft["지도 미세 조정(SFT)<br>지시 데이터셋으로 추가 학습"]
  sft --> rlhf["RLHF<br>사람 피드백 기반 강화학습"]
  rlhf --> aligned["정렬된 모델<br>(예: ChatGPT)"]
</div>

<ul>
  <li><strong>지도 미세 조정(SFT)</strong>: 정렬의 핵심 과정. 사용자의 요청·지시와 그에 대한 적절한 응답을 정리한 <strong>지시 데이터셋</strong>으로 모델을 추가 학습시킨다.</li>
  <li><strong>RLHF</strong>: 사람의 피드백(어떤 답변이 더 나은지에 대한 선호)을 보상 신호로 활용해 모델을 강화 학습시키는 단계. SFT만으로 다 잡지 못하는 답변의 품질·안전성을 추가로 조정한다.</li>
</ul>

## AI 에이전트를 위한 평가(Evals) ② — 리서치/컴퓨터 사용 에이전트와 평가 구축 로드맵


<p>[[2026-06-19]]에서 정리한 평가의 기본 구조와 코딩/대화형 에이전트 평가에 이어, 같은 자료의 나머지 부분 — 리서치·컴퓨터 사용 에이전트 평가, 평가의 비결정성, 그리고 평가를 0에서 1로 만드는 로드맵 — 을 정리한다.</p>

### 리서치 에이전트 평가


<p>리서치 에이전트는 정보를 수집·종합·분석해 답변이나 보고서를 산출한다. 코딩 에이전트는 단위 테스트라는 이진(통과/실패) 신호를 갖지만, 리서치 품질은 오직 과제에 상대적으로만 판단할 수 있다. 무엇이 "포괄적"인지, "출처가 충실"한지, "정확"한지조차 맥락에 따라 달라진다 — 시장 스캔, 인수를 위한 실사, 과학 보고서는 각기 다른 기준을 요구한다.</p>

<ul>
  <li>전문가들조차 종합 결과가 포괄적인지에 대해 의견이 다를 수 있다.</li>
  <li>참조 콘텐츠가 끊임없이 변하기 때문에 정답(ground truth) 자체가 변동한다.</li>
  <li>더 길고 개방적인 결과물은 실수가 들어갈 여지를 더 많이 만든다.</li>
</ul>

<p>이런 어려움을 다루는 전략은 여러 채점자 유형을 결합하는 것이다.</p>

<div class="mermaid">
graph LR
  out["리서치 결과물"] --> g["근거성(groundedness)<br>검사 — 주장이 출처로 뒷받침되는가"]
  out --> c["커버리지 검사<br>좋은 답변이 포함해야 할 핵심 사실"]
  out --> s["출처 품질 검사<br>권위 있는 출처인가(첫 검색 결과가 아니라)"]
</div>

<p>리서치 품질의 주관적 성격을 고려하면, 이런 에이전트를 효과적으로 채점하려면 LLM 기반 루브릭을 전문가의 사람 판단에 맞추어 자주 보정해야 한다.</p>

### 컴퓨터 사용 에이전트 평가


<p>컴퓨터 사용 에이전트는 API나 코드 실행이 아니라, 인간과 동일한 인터페이스 — 스크린샷, 마우스 클릭, 키보드 입력, 스크롤 — 를 통해 소프트웨어와 상호작용한다. 브라우저 사용 에이전트는 토큰 효율성과 지연 시간 사이의 트레이드오프를 갖는다: DOM 기반 상호작용은 빠르지만 토큰을 많이 쓰고, 스크린샷 기반 상호작용은 느리지만 토큰 효율이 더 높다.</p>

### 평가에서 비결정성 다루기


<img src="/learning-notes/images/2026-06-22-1.png" alt="에이전트 평가의 비결정성">

<p>에이전트는 같은 입력에도 매번 다른 경로로 과제를 풀 수 있다. 결과(outcome)를 채점하면 경로의 다양성을 허용하면서도 성공 여부를 판단할 수 있고, 트랜스크립트(경로) 자체를 너무 엄격하게 채점하면 정답이지만 예상과 다른 해법을 부당하게 실패로 처리하게 된다.</p>

### 평가 구축 로드맵: 0에서 1까지


<p>평가 주도(eval-driven) 에이전트 개발의 로드맵은 성공을 일찍 정의하고, 명확하게 측정하며, 지속적으로 반복하는 것이다.</p>

<details>
<summary>각 단계 상세</summary>
<ul>
  <li><strong>Step 0. 일찍 시작하기</strong>: 평가는 미루는 시간이 길어질수록 구축하기 어려워진다. 수백 개의 과제가 필요하다고 생각해 미루는 경우가 많지만, 실제 실패 사례에서 가져온 20~50개의 단순한 과제만으로도 훌륭한 출발이 된다. 이미 프로덕션 단계라면 버그 트래커와 지원 대기열을 살펴본다.</li>
  <li><strong>Step 1. 이미 수동으로 테스트하던 것부터 시작</strong>: 개발 중에 실행하는 수동 점검 — 각 출시 전에 검증하는 행동들과 최종 사용자가 흔히 시도하는 과제들 — 부터 시작한다. 이미 프로덕션 단계라면 버그 트래커와 지원 대기열을 살펴본다. 사용자가 보고한 실패를 테스트 케이스로 전환하면 평가 스위트가 실제 사용 패턴을 반영하게 되고, 사용자 영향도에 따라 우선순위를 정하면 노력을 의미 있는 곳에 투입할 수 있다.</li>
  <li><strong>Step 2. 모호하지 않은 과제를 참조 해법과 함께 작성</strong>: 각 과제에 대해 참조 해법(모든 채점자를 통과하는 작동 가능한 출력)을 만들어 둔다. 좋은 과제란 두 명의 도메인 전문가가 독립적으로 동일한 통과/실패 판정에 도달하는 과제다 — 그들이 직접 과제를 통과할 수 없다면 다시 다듬어야 하고, 과제 명세의 모호함은 지표상의 잡음이 된다. 모델 기반 채점자의 루브릭도 마찬가지로, 채점자가 확인하는 모든 것은 과제 설명에서 명확히 드러나야 하며 에이전트가 모호한 명세 때문에 실패해서는 안 된다. 각 과제는 지시를 올바르게 따르는 에이전트라면 통과할 수 있어야 한다.</li>
  <li><strong>Step 3. 균형 잡힌 문제 집합 구성</strong>: 어떤 행동이 일어나야 하는 경우와 일어나지 않아야 하는 경우를 모두 테스트한다. 예를 들어 검색해야 할 때 검색하는지만 테스트하면, 거의 모든 것에 대해 검색을 하는 에이전트를 얻게 될 수 있다.</li>
  <li><strong>Step 4. 안정적인 환경을 갖춘 견고한 평가 하니스 구축</strong>: 평가에 쓰이는 에이전트가 프로덕션 에이전트와 동일하게 작동해야 하고, 각 트라이얼은 깨끗한 환경에서 시작해 "격리"되어야 한다. 실행 간에 공유되는 상태(남은 파일, 캐시, 자원 고갈)는 에이전트 성능이 아닌 인프라 불안정성으로 인한 실패를 일으키거나 성능을 인위적으로 부풀릴 수 있다.</li>
  <li><strong>Step 5. 채점자를 신중하게 설계</strong>: 가능한 곳에는 결정론적 채점자, 유연성이 필요한 곳에는 LLM 채점자, 추가 검증에는 사람 채점자를 쓴다. 창의성을 불필요하게 처벌하지 않으려면 에이전트가 거친 경로보다 결과물을 채점하는 것이 보통 더 낫고, 여러 구성요소로 이루어진 과제에는 부분 점수를 둔다. LLM 심사자는 사람 전문가와 긴밀히 보정해야 하며, 정보가 부족할 때는 "알 수 없음"을 반환하도록 빠져나갈 길을 줘서 환각을 줄인다. 한 LLM이 모든 차원을 채점하기보다 차원별로 분리된 LLM 심사자를 쓰는 게 도움이 될 수 있고, 채점자는 우회나 부정행위에 저항력을 가져야 한다.</li>
  <li><strong>Step 6. 트랜스크립트 직접 확인</strong>: 점수가 오르지 않을 때, 그것이 평가 자체의 문제가 아니라 에이전트의 성능 때문이라는 확신이 있어야 한다. 많은 트라이얼의 트랜스크립트와 점수를 직접 읽지 않으면 채점자가 잘 작동하는지 알 수 없다 — 트랜스크립트는 에이전트가 진짜 실수를 했는지, 채점자가 유효한 해법을 잘못 거부했는지를 알려준다.</li>
  <li><strong>Step 7. 역량 평가 포화 감지</strong>: 100%에 도달한 평가는 퇴보를 추적할 뿐 개선 신호는 주지 못한다. 평가 포화는 에이전트가 풀 수 있는 모든 과제를 통과해 더 이상 개선 여지가 없을 때 발생한다. 누군가 평가의 세부사항을 파고들고 트랜스크립트를 읽어보기 전까지는 점수를 액면 그대로 받아들이지 않아야 한다.</li>
  <li><strong>Step 8. 평가 스위트를 장기적으로 건강하게 유지</strong>: 평가 주도 개발 — 에이전트가 어떤 능력을 수행할 수 있기 전에 그 계획된 역량을 정의하는 평가를 먼저 만들고, 에이전트가 잘 수행할 때까지 반복한다.</li>
</ul>
</details>

### 평가는 다른 방법들과 어떻게 어울리는가


<p>자동화된 평가는 프로덕션에 배포하거나 실제 사용자에게 영향을 미치지 않고도 수천 개의 과제에 대해 에이전트를 실행할 수 있다. 완전한 그림을 얻으려면 프로덕션 모니터링, 사용자 피드백, A/B 테스트, 수동 트랜스크립트 검토, 체계적인 사람 평가까지 함께 봐야 한다.</p>

<img src="/learning-notes/images/2026-06-22-3.png" alt="평가와 다른 방법의 관계 1">
<img src="/learning-notes/images/2026-06-22-4.png" alt="평가와 다른 방법의 관계 2">

<p>안전 공학의 스위스 치즈 모델처럼, 단 하나의 평가 계층만으로는 모든 문제를 포착할 수 없다 — 여러 방법을 함께 쓰면 한 계층을 통과한 실패도 다른 계층에서 포착된다.</p>

<img src="/learning-notes/images/2026-06-22-5.png" alt="평가와 다른 방법의 관계 3 — 스위스 치즈 모델">

<ul>
  <li><strong>자동화된 평가</strong>: 출시 전 단계와 CI/CD에서 특히 유용하며, 각 변경 사항과 모델 업그레이드마다 실행되는 1차 방어선이다.</li>
  <li><strong>프로덕션 모니터링</strong>: 출시 이후에 작동해 분포 변화(drift)와 예상치 못한 실제 환경 실패를 감지한다.</li>
  <li><strong>A/B 테스트</strong>: 충분한 트래픽이 확보된 후 중요한 변경 사항을 검증한다.</li>
  <li><strong>사용자 피드백 + 트랜스크립트 검토</strong>: 그 사이의 공백을 메우는 지속적인 실천 — 피드백을 끊임없이 분류하고, 매주 트랜스크립트 일부를 샘플링해 읽고, 필요할 때 더 깊이 판다.</li>
  <li><strong>체계적인 사람 연구</strong>: LLM 채점자를 보정하거나, 사람의 합의가 기준이 되는 주관적 출력을 평가할 때를 위해 남겨둔다.</li>
</ul>

### 정리


<p>평가가 없는 팀은 반응적인 루프에 갇힌다 — 하나의 실패를 고치면 또 다른 실패가 생기고, 진짜 퇴보와 잡음을 구분할 수 없다. 평가는 팀 전체에게 명확히 올라야 할 산을 제시하며, "에이전트가 더 나빠진 것 같다"는 느낌을 실행 가능한 무언가로 바꾸어준다. 패턴은 에이전트 유형에 따라 다르지만 기본 원칙은 변하지 않는다.</p>

<ul>
  <li>일찍 시작하고 완벽한 스위트를 기다리지 않는다.</li>
  <li>실제로 마주치는 실패에서 현실적인 과제를 끌어낸다.</li>
  <li>명확하고 견고한 성공 기준을 정의한다.</li>
  <li>채점자를 신중하게 설계하고 여러 유형을 결합한다.</li>
  <li>문제가 모델에게 충분히 어려운지 확인한다.</li>
  <li>평가의 신호 대 잡음 비율을 개선하기 위해 반복한다.</li>
  <li>트랜스크립트를 읽는다.</li>
</ul>
