## 멀티 에이전트 리서치 시스템을 설계할 때 기억할 것

이 글에서 가져갈 핵심은 "에이전트를 여러 개 쓰면 더 똑똑해진다"가 아니다. 더 정확히는 **컨텍스트와 탐색 예산을 여러 독립 단위로 쪼개고, 마지막에 압축해서 합치는 구조가 리서치 작업에 잘 맞는다**는 것이다.

단일 에이전트는 한 컨텍스트 안에서 문제를 이해하고, 검색하고, 판단하고, 답변까지 해야 한다. 질문이 넓어질수록 이 구조는 금방 병목이 생긴다. 반면 멀티 에이전트 구조는 탐색 자체를 병렬화한다. 여러 서브 에이전트가 서로 다른 가설, 출처, 하위 질문을 조사하고 리드 에이전트는 그 결과를 종합한다.

### 내 해석: 멀티 에이전트는 병렬 검색이 아니라 병렬 압축이다

리서치에서 가장 비싼 일은 정보를 "찾는 것" 자체보다, 많은 후보 정보 중 무엇이 중요한지 걸러내는 일이다. 그래서 멀티 에이전트의 진짜 역할은 검색 요청을 여러 개 날리는 것이 아니라, 각자 작은 리서처처럼 움직이며 자기 영역의 정보를 압축해 오는 것이다.

<div class="mermaid">
flowchart LR
    Q[사용자 질문] --> L[리드 에이전트]
    L --> A[서브 에이전트 A<br/>시장/기업 탐색]
    L --> B[서브 에이전트 B<br/>기술/제품 탐색]
    L --> C[서브 에이전트 C<br/>근거/출처 검증]
    A --> SA[압축된 발견]
    B --> SB[압축된 발견]
    C --> SC[압축된 발견]
    SA --> L
    SB --> L
    SC --> L
    L --> R[종합 답변]
</div>

이 구조에서 리드 에이전트의 책임은 답을 직접 다 찾는 것이 아니라 다음을 판단하는 것이다.

- 질문을 어떤 하위 문제로 나눌 것인가
- 어떤 하위 문제는 넓게 훑고, 어떤 하위 문제는 깊게 파고들 것인가
- 서브 에이전트 결과 중 무엇을 믿고 무엇을 버릴 것인가
- 부족한 근거가 있으면 추가 탐색을 시킬 것인가

즉, 리드 에이전트는 "검색하는 사람"보다 "편집장"에 가깝다.

### 언제 멀티 에이전트를 쓸 만한가

멀티 에이전트는 비용이 크다. 토큰을 많이 쓰고, 도구 호출도 늘고, 디버깅할 상태도 많아진다. 그래서 기본 선택지로 두면 과하다. 다음 조건이 맞을 때만 이 구조가 값어치를 한다.

| 상황 | 멀티 에이전트 적합도 | 이유 |
| --- | --- | --- |
| 질문이 좁고 답이 명확함 | 낮음 | 단일 검색/단일 에이전트가 빠르고 싸다 |
| 여러 출처를 비교해야 함 | 높음 | 출처별/관점별로 탐색을 나누기 좋다 |
| 탐색 중 방향 전환이 필요함 | 높음 | 중간 발견을 바탕으로 후속 질문을 만들 수 있다 |
| 정답보다 조사 품질이 중요함 | 높음 | 근거 수집, 반례 탐색, 교차 검증을 분리할 수 있다 |
| 응답 지연과 비용에 민감함 | 낮음 | 병렬화해도 전체 토큰 비용은 크게 늘어난다 |

내 기준으로는 **"하위 질문을 3개 이상 자연스럽게 만들 수 있고, 각 하위 질문의 답이 서로 독립적으로 조사 가능할 때"** 멀티 에이전트를 고려할 만하다.

### Research 아키텍처에서 배울 점

원문에서 소개한 Research 시스템은 오케스트레이터-워커 구조다. 리드 에이전트가 계획을 세우고, 서브 에이전트들이 병렬로 조사한 뒤, 다시 리드 에이전트가 결과를 종합한다.

<img src="/learning-notes/images/2026-06-29-1.png" alt="Research의 멀티 에이전트 아키텍처">

중요한 차이는 기존 RAG와의 비교에서 드러난다.

기존 RAG는 보통 한 번 검색한 문서 조각을 바탕으로 답한다. 이 방식은 질문이 명확하고 필요한 정보가 문서 안에 잘 들어 있을 때 강하다. 하지만 리서치 질문은 다르다. 처음 검색한 결과가 다음 질문을 바꾸고, 새로 발견한 이름이나 사건이 다시 검색 키워드가 된다.

<img src="/learning-notes/images/2026-06-29-2.png" alt="정적 검색과 동적 멀티 에이전트 검색의 차이">

따라서 멀티 에이전트 리서치는 RAG의 대체라기보다, **정적 검색을 동적 탐색 루프로 확장한 형태**로 이해하는 편이 낫다.

### 실패 모드

멀티 에이전트 시스템의 어려움은 모델 성능보다 조율에서 많이 나온다. 원문에서 언급된 실패를 일반화하면 다음 네 가지다.

| 실패 | 증상 | 원인 | 대응 |
| --- | --- | --- | --- |
| 과도한 분기 | 간단한 질문에도 서브 에이전트를 너무 많이 만듦 | 위임 기준이 없음 | 질의 복잡도별 에이전트 수 제한 |
| 무한 탐색 | 없는 출처나 희박한 단서를 계속 찾음 | 중단 조건이 없음 | 실패/불확실성 보고를 허용 |
| 중복 조사 | 여러 에이전트가 같은 내용을 반복 조사 | 역할 경계가 흐림 | 서브태스크를 출처/관점/가설 기준으로 분리 |
| 조율 소음 | 업데이트가 너무 많아 리드 에이전트 컨텍스트가 오염됨 | 보고 형식이 없음 | 결과는 짧게, 근거는 참조로 전달 |

여기서 배울 점은 프롬프트를 "열심히 조사해" 수준으로 두면 안 된다는 것이다. 에이전트에게는 역할, 예산, 중단 조건, 보고 형식을 같이 줘야 한다.

### 프롬프트는 규칙보다 휴리스틱을 담아야 한다

에이전트 프롬프트를 엄격한 if-else 규칙처럼 만들면 실제 리서치의 변동성을 감당하기 어렵다. 더 중요한 것은 좋은 리서처의 판단 기준을 심는 것이다.

예를 들면 다음과 같은 휴리스틱이 필요하다.

- 처음에는 넓게 훑고, 유망한 방향이 보이면 좁혀라.
- 출처가 약하면 결론을 강하게 말하지 마라.
- 새로운 고유명사, 날짜, 수치가 나오면 교차 검증하라.
- 같은 정보가 반복되면 더 찾지 말고 요약하라.
- 실패한 검색어를 반복하지 말고 검색 전략을 바꿔라.
- 불확실한 내용은 추정과 확인된 사실을 분리해서 보고하라.

특히 오케스트레이터 프롬프트에는 "어떻게 답할지"보다 "어떻게 일을 나눌지"가 중요하다. 서브 에이전트 프롬프트에는 "무엇을 조사할지"보다 "무엇을 가져오고 무엇은 버릴지"가 중요하다.

### 도구 설계는 에이전트의 행동 반경을 정한다

멀티 에이전트 시스템에서는 도구 설명이 곧 UI다. 사람이 버튼 이름과 설명을 보고 기능을 이해하듯, 에이전트도 도구 이름과 스키마를 보고 사용할지 말지를 판단한다.

좋은 도구는 다음 성질을 가져야 한다.

- 목적이 하나로 좁혀져 있다.
- 입력 스키마가 모호하지 않다.
- 실패했을 때 이유를 알려준다.
- 너무 많은 결과를 한 번에 반환하지 않는다.
- 후속 탐색에 필요한 참조를 안정적으로 제공한다.

도구가 애매하면 에이전트는 불필요하게 많이 호출하거나, 잘못된 상황에서 사용하거나, 실패를 복구하지 못한다. 결국 프롬프트 품질과 도구 품질은 따로 갈 수 없다.

### 평가는 "정해진 경로"가 아니라 "좋은 결과와 합리적 과정"을 본다

에이전트는 같은 입력에서도 매번 다른 경로를 택할 수 있다. 그래서 일반적인 테스트처럼 "이 순서로 이 도구를 호출해야 한다"를 강하게 요구하면 좋은 행동까지 실패로 판정할 수 있다.

대신 평가는 세 층으로 나누는 편이 맞다.

<div class="mermaid">
flowchart TD
    A[최종 답변 평가] --> B[근거 평가]
    B --> C[탐색 과정 평가]
    C --> D[시스템 상호작용 평가]
</div>

- 최종 답변 평가: 질문에 답했는가, 중요한 누락이 없는가
- 근거 평가: 출처가 충분하고 결론을 뒷받침하는가
- 탐색 과정 평가: 합리적인 하위 질문을 세웠는가
- 시스템 상호작용 평가: 에이전트 간 중복, 과소/과잉 위임, 무한 탐색이 없었는가

LLM-as-judge는 이 중 일부를 자동화할 수 있지만, 에이전트 시스템에서는 사람 평가가 계속 필요하다. 특히 "겉보기엔 그럴듯하지만 조사 방향 자체가 틀린 경우"는 자동 평가가 놓치기 쉽다.

### 운영 관점에서 더 어려워지는 부분

멀티 에이전트는 데모보다 운영이 어렵다. 이유는 상태와 오류가 누적되기 때문이다.

단일 요청-응답 시스템에서는 실패하면 다시 시도하면 된다. 하지만 장기 실행 에이전트는 이미 여러 도구를 호출했고, 중간 산출물을 만들었고, 다른 에이전트의 결과를 기다리고 있을 수 있다. 이 상태에서 단순 재시작은 비용도 크고 맥락도 잃는다.

운영 시스템에는 최소한 다음 장치가 필요하다.

- 에이전트별 작업 상태 저장
- 도구 실패를 구조화해서 에이전트에게 전달
- 중간 산출물을 파일이나 DB 같은 외부 저장소에 남기기
- 리드 에이전트가 참조만 받아도 결과를 추적할 수 있는 식별자 설계
- 프롬프트 변경 전후 행동을 비교할 수 있는 관찰 가능성

특히 좋은 패턴은 서브 에이전트가 모든 결과를 대화로 넘기지 않고, 산출물을 외부에 저장한 뒤 참조만 전달하는 것이다. 코드, 보고서, 시각화처럼 구조화된 결과는 리드 에이전트를 거치며 말로 재전달될수록 품질이 떨어진다.

### 내가 적용한다면

내가 리서치용 멀티 에이전트를 만든다면 처음부터 거대한 범용 시스템을 만들지는 않을 것이다. 먼저 세 역할만 둔다.

| 역할 | 책임 | 출력 |
| --- | --- | --- |
| Planner | 질문 분해, 탐색 예산 배분 | 하위 질문 목록 |
| Researcher | 하위 질문 조사, 근거 수집 | 짧은 발견 요약 + 출처 |
| Synthesizer | 충돌 해결, 최종 답변 작성 | 답변 + 불확실성 |

그리고 각 실행마다 다음을 기록한다.

- 어떤 하위 질문을 만들었는가
- 어떤 도구를 몇 번 호출했는가
- 어떤 출처를 채택/기각했는가
- 최종 답변에서 어떤 근거가 쓰였는가
- 사람이 보기엔 과잉 탐색이나 누락이 있었는가

이 로그가 있어야 프롬프트를 고쳐도 실제로 좋아졌는지 판단할 수 있다.

### 정리

멀티 에이전트 리서치 시스템은 "에이전트를 많이 붙이는 기술"이 아니라, 복잡한 리서치 문제를 여러 컨텍스트에서 병렬로 탐색하고 다시 압축하는 설계다. 효과가 큰 문제는 넓고 모호하며, 중간 발견에 따라 다음 질문이 바뀌는 문제다.

반대로 질문이 작거나 비용이 중요한 경우에는 과한 구조다. 적용 여부는 멋진 아키텍처가 아니라 탐색 폭, 검증 필요성, 비용 대비 가치로 판단해야 한다.
