## 멀티 에이전트 리서치 시스템

멀티 에이전트 시스템은 여러 에이전트가 협력해 하나의 작업을 수행하는 구조다. 여기서 에이전트는 LLM이 도구를 반복적으로 사용하며 자율적으로 판단하고 실행하는 단위로 볼 수 있다.

핵심 전제는 단순하다. 범용 지능을 가진 에이전트라도 혼자 긴 리서치 작업을 처리할 때는 컨텍스트, 탐색 폭, 도구 사용량의 한계에 부딪힌다. 여러 에이전트가 각자 독립적인 컨텍스트와 탐색 경로를 가지면, 더 넓은 문제 공간을 동시에 조사하고 중요한 정보만 압축해 리드 에이전트에게 전달할 수 있다.

### 왜 멀티 에이전트인가

멀티 에이전트 시스템이 효과적인 가장 큰 이유는 문제 해결에 사용할 수 있는 토큰과 탐색 경로를 확장해 준다는 점이다.

리서치 작업은 단순한 one-shot 파이프라인으로 처리하기 어렵다. 조사 중에 새로운 방향이 열리고, 주변 주제를 탐색해야 하며, 중간 결과를 바탕으로 다음 검색 전략을 바꿔야 하기 때문이다. 이때 검색은 방대한 정보 집합에서 핵심 통찰만 추려내는 압축 과정이 된다.

서브 에이전트는 이 압축 과정을 병렬화한다.

- 각 에이전트는 독립적인 컨텍스트 윈도우를 가진다.
- 서로 다른 도구, 프롬프트, 탐색 경로를 사용할 수 있다.
- 질문의 여러 측면을 동시에 탐색한 뒤 핵심 정보만 리드 에이전트에게 전달한다.
- 특정 탐색 경로에 과도하게 의존하는 문제를 줄인다.

내부 평가에서는 여러 독립적인 방향을 동시에 탐색해야 하는 너비 우선 질의에서 멀티 에이전트 리서치 시스템이 특히 강했다. 성능 차이의 상당 부분은 토큰 사용량으로 설명되었고, 나머지는 도구 호출 횟수와 모델 선택의 영향을 받았다.

다만 비용도 크다. 일반적인 에이전트가 채팅보다 약 4배 많은 토큰을 사용했다면, 멀티 에이전트 시스템은 채팅보다 약 15배 많은 토큰을 사용했다. 따라서 멀티 에이전트 구조는 추가 비용을 정당화할 만큼 가치가 높고 복잡한 작업에 사용하는 것이 맞다.

### Research 아키텍처

Research 시스템은 오케스트레이터-워커 패턴을 사용한다. 리드 에이전트가 전체 과정을 조율하고, 병렬로 동작하는 전문화된 서브 에이전트에게 작업을 분배한다.

<img src="/learning-notes/images/2026-06-29-1.png" alt="Research의 멀티 에이전트 아키텍처">

흐름은 다음과 같다.

1. 사용자의 질문이 리드 에이전트로 들어온다.
2. 리드 에이전트가 질문을 분석하고 탐색 전략을 세운다.
3. 서로 다른 측면을 조사할 서브 에이전트들을 생성한다.
4. 서브 에이전트들은 검색 도구를 반복적으로 사용하며 정보를 수집한다.
5. 각 서브 에이전트가 중요한 결과를 압축해 리드 에이전트에게 전달한다.
6. 리드 에이전트가 결과를 종합해 최종 답변을 만든다.

이 구조에서 서브 에이전트는 단순한 검색기가 아니라 지능형 필터에 가깝다. 예를 들어 2025년의 AI 에이전트 기업을 조사한다면, 여러 서브 에이전트가 각기 다른 범주나 출처를 탐색하고 기업 목록과 근거를 리드 에이전트에게 전달한다.

기존 RAG는 입력 질문과 유사한 문서 조각을 한 번 검색한 뒤 그 정보로 답변을 생성하는 정적 검색에 가깝다. 반면 멀티 에이전트 리서치는 여러 단계의 검색을 수행하고, 새 정보를 발견할 때마다 탐색 방향을 동적으로 조정한다.

### 프롬프트와 도구 설계

멀티 에이전트 시스템은 단일 에이전트 시스템보다 조율 복잡도가 훨씬 빠르게 증가한다. 초기 버전의 에이전트들은 간단한 질문에도 서브 에이전트 50개를 생성하거나, 존재하지 않는 출처를 찾기 위해 웹을 끝없이 뒤지거나, 과도한 업데이트로 서로의 주의를 분산시키는 실수를 했다.

각 에이전트는 프롬프트에 의해 제어되므로, 행동 개선의 핵심 수단은 프롬프트 엔지니어링이었다. 중요한 방향은 딱딱한 규칙을 강제하는 것이 아니라 좋은 휴리스틱을 심는 것이다.

효과적인 프롬프트에는 숙련된 리서처의 작업 방식이 녹아 있어야 한다.

- 어려운 질문을 작은 작업으로 나누기
- 출처 품질을 신중하게 평가하기
- 새 정보에 따라 검색 방식을 조정하기
- 깊이 파고들 때와 넓게 탐색할 때를 구분하기
- 오케스트레이터가 언제, 어떻게 위임해야 하는지 알려주기
- 질의 복잡도에 맞춰 노력과 도구 호출량을 조절하기

도구 설계도 프롬프트만큼 중요하다. 에이전트와 도구 사이의 인터페이스는 사람과 컴퓨터 사이의 인터페이스만큼 중요하므로, 도구는 목적이 분명하고 설명이 이해하기 쉬워야 한다. 좋은 도구 설명은 에이전트가 어떤 상황에서 어떤 도구를 써야 하는지 판단하게 해 준다.

### 에이전트 평가

신뢰할 수 있는 AI 애플리케이션을 만들려면 평가가 필수이고, 에이전트도 예외가 아니다. 다만 에이전트 평가는 일반적인 결정론적 소프트웨어 평가와 다르다.

기존 평가는 입력 X가 주어지면 시스템이 경로 Y를 거쳐 결과 Z를 만들어야 한다고 가정하는 경우가 많다. 하지만 에이전트는 같은 출발점에서 서로 다른 경로를 선택할 수 있고, 그 경로들이 모두 합리적일 수 있다. 따라서 특정 절차를 그대로 따랐는지만 검사하는 방식으로는 충분하지 않다.

에이전트 평가는 "합리적인 과정을 거쳐 올바른 결과를 얻었는가"를 볼 수 있어야 한다.

- 작은 샘플이라도 즉시 평가를 시작한다.
- LLM-as-judge는 잘 설계하면 확장 가능한 평가 수단이 된다.
- 자동 평가가 놓치는 문제를 잡기 위해 사람의 수동 평가가 여전히 필요하다.
- 개별 에이전트뿐 아니라 에이전트 사이의 상호작용 패턴도 관찰해야 한다.

멀티 에이전트 시스템은 명시적으로 프로그래밍하지 않은 창발적 행동을 보인다. 리드 에이전트 프롬프트를 조금 바꿨을 뿐인데 서브 에이전트의 행동이 예측하기 어려운 방식으로 달라질 수 있다. 그래서 좋은 프롬프트는 단순한 지시문이 아니라 역할 분담, 문제 해결 방식, 자원 배분을 정의하는 협업 프레임워크에 가깝다.

### 운영 신뢰성

에이전트 시스템에서는 작은 변경이 연쇄적으로 큰 행동 변화를 일으킨다. 특히 오래 상태를 유지하며 많은 도구 호출을 수행하는 에이전트는 오류가 누적되기 쉽다.

한 번 오류가 발생했다고 전체 작업을 처음부터 다시 시작할 수는 없다. 재시작은 비용이 크고 사용자 경험도 나쁘다. 대신 시스템은 중간 오류를 에이전트에게 알려주고, 에이전트가 다른 방법으로 적응할 수 있게 만들어야 한다. 실제로 특정 도구가 실패하고 있다는 정보를 알려주면 에이전트가 스스로 대안을 찾는 경우가 효과적이었다.

디버깅도 어렵다. 에이전트는 동일한 프롬프트를 사용해도 실행할 때마다 다른 결정을 내릴 수 있는 비결정적 특성을 가진다. 그래서 일반적인 로그나 메트릭을 넘어, 에이전트의 의사결정 패턴과 상호작용 구조를 관찰할 수 있는 고수준의 observability가 필요하다.

프로덕션 수준의 멀티 에이전트 시스템에는 다음 요소가 필요하다.

- 실패를 견디는 상태 관리
- 도구 실패를 에이전트에게 전달하는 오류 처리
- 에이전트 간 상호작용을 볼 수 있는 관찰 가능성
- 프롬프트와 도구 변경에 대한 신중한 배포
- 병렬 실행으로 병목을 줄이는 실행 구조

### 긴 대화와 산출물 관리

프로덕션 환경의 에이전트는 수백 턴에 걸친 대화를 수행할 수 있으므로 컨텍스트 관리가 중요하다. 컨텍스트 한계에 가까워지면 새 컨텍스트를 가진 서브 에이전트를 만들고, 이전 작업을 요약해 외부 메모리에 저장한 뒤 이어가는 패턴을 사용할 수 있다.

또 하나의 중요한 패턴은 서브 에이전트가 결과를 리드 에이전트에게 모두 말로 전달하지 않게 하는 것이다. 코드, 보고서, 데이터 시각화처럼 구조화된 결과물은 전문화된 서브 에이전트가 파일 시스템 같은 외부 시스템에 직접 저장하고, 리드 에이전트에게는 가벼운 참조만 전달하는 편이 낫다.

이 방식은 리드 에이전트를 거치며 정보가 손실되는 문제를 줄이고, 산출물의 품질도 높인다.

### 정리

멀티 에이전트 리서치 시스템의 핵심은 병렬 추론과 정보 압축이다. 여러 에이전트가 독립적으로 탐색하고, 리드 에이전트가 그 결과를 종합하면 복잡하고 넓은 리서치 질의에서 강한 성능을 낼 수 있다.

하지만 토큰 비용과 운영 복잡도도 함께 증가한다. 따라서 멀티 에이전트 시스템은 모든 문제의 기본값이 아니라, 높은 가치와 높은 복잡도를 가진 작업에 선택적으로 적용해야 한다. 실제 서비스로 만들려면 프롬프트 설계, 도구 설계, 평가, 관찰 가능성, 오류 처리, 배포 전략까지 모두 함께 다뤄야 한다.
